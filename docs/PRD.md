## 1. 개요

### 1.1 배경
일상 업무에서 사용하는 자원(사내 웹 시스템, SaaS, 로컬 설치 프로그램)이 다음과 같이 여러 경로에 흩어져 있어 접근 비용이 크다.

- 업무 시스템 URL → Edge / Chrome 북마크
- 자주 쓰는 프로그램 → Windows 작업 표시줄에 고정된 바로가기 폴더
- 카테고리 분류 → 브라우저 북마크 폴더 / 작업 표시줄 폴더에 분산

결과적으로 "어디에 있었지?"를 찾는 시간이 누적되고, 브라우저 종류가 바뀌거나 PC를 교체할 때마다 재구성이 필요하다.

### 1.2 목적
**모든 업무 자원(웹 URL + 로컬 프로그램)을 카테고리별로 한 페이지에서 시각적으로 파악하고, 단 한 번의 클릭으로 실행**할 수 있는 개인용 Workbench를 제공한다.

### 1.3 제품 비전 (1-liner)
> "내 업무의 홈 화면 — 모든 도구를 한 화면, 한 번의 클릭으로."

### 1.4 성공 지표 (KPI 후보)
- 평균 자원 접근 시간: **3초 이내** (앱 실행 → 아이템 클릭 → 대상 오픈)
- 사용자 1명이 관리 가능한 아이템 수: **300개 이상**에서 체감 지연 없음
- 편집 모드에서 드래그 앤 드롭 1회 조작의 오류율 < 1%

## 2. 대상 사용자 및 사용 맥락

### 2.1 1차 사용자 (Primary Persona)
- 사내 여러 시스템(사내 포털, 이슈 트래커, Wiki, 모니터링, 개발 도구 등)을 하루 종일 넘나드는 **지식 노동자 / 엔지니어**
- 로컬에 설치된 IDE, 터미널, 메신저, Office 제품군도 함께 사용
- Windows 환경 중심, 관리자 권한이 없는 표준 사내 PC일 수 있음

### 2.2 대표 사용자 스토리
- **US-01** 나는 아침마다 Workbench를 열어 오늘 사용할 모든 시스템을 한눈에 확인하고 싶다.
- **US-02** 나는 "개발", "사내 포털", "문서" 같은 카테고리 단위로 자원을 묶어 탐색하고 싶다.
- **US-03** 나는 새 시스템이 생기면 즉시 Workbench에 추가하고, 불필요한 것은 바로 지우고 싶다.
- **US-04** 나는 자주 쓰는 것을 드래그로 맨 앞/맨 위로 옮겨 접근성을 높이고 싶다.
- **US-05** 나는 PC를 바꾸거나 재설치할 때 설정을 그대로 가져가고 싶다.

## 3. 범위

### 3.1 In-Scope (v1.0)
- 단일 사용자, 로컬 실행형 Windows 데스크톱 애플리케이션
- 웹 URL 및 로컬 실행 파일(.exe / .lnk / UWP `shell:AppsFolder`) 바로가기 실행
- 카테고리 기반 구성 및 편집 모드 제공
- 설정/데이터의 파일 기반 저장 및 수동 내보내기/가져오기

### 3.2 Out-of-Scope (v1.0)
- 다중 사용자 / 서버 기반 동기화 (클라우드 sync는 v2 이후 검토)
- 모바일/웹 버전
- 자동 북마크 동기화 (Edge/Chrome 북마크 실시간 연동)
- 팀 공유 / 권한 관리
- macOS, Linux 지원

## 4. 용어 정의

| 용어 | 정의 |
|---|---|
| **섹션 (Section)** | 최상위 구획. 예: "업무 시스템", "개발 도구", "개인" 등. 한 페이지 내 가로 또는 세로로 배치 |
| **카테고리 (Category)** | 섹션 하위 분류. 예: 업무 시스템 > "사내 포털", "이슈 트래커" |
| **아이템 (Item)** | 카테고리 내 실제 실행 가능한 바로가기. 웹 URL 또는 로컬 프로그램 |
| **편집 모드 (Edit Mode)** | 섹션/카테고리/아이템을 추가·수정·삭제·재배치할 수 있는 토글 상태 |
| **실행 모드 (Launch Mode)** | 기본 상태. 아이템 클릭 시 즉시 실행만 수행 |

계층 구조:
```
Page
└── Section
    └── Category
        └── Item (Web URL | Local App)
```

## 5. 기능 요구사항

### 5.1 사용자 명시 기본 기능 (필수)

#### FR-1. 한 페이지, 카테고리별 구성
- 한 화면에 다수의 섹션/카테고리/아이템을 동시에 표시한다.
- 카테고리는 격자(grid) 형태로 배치되며, 화면 크기에 반응형으로 재배치된다.
- 아이템은 각각 아이콘 + 레이블로 타일 형태로 표시된다.

#### FR-2. 아이콘 처리
- **로컬 프로그램 아이템**: 대상 파일(.exe/.lnk/AppUserModelID)에서 **아이콘을 자동 추출**하여 사용한다.
  - 추출 실패 시 기본 아이콘으로 폴백하고, 수동 지정도 가능해야 한다.
- **웹 URL 아이템**: 사용자가 **아이콘을 검색 또는 지정**할 수 있다.
  - (a) 대상 URL의 **favicon 자동 조회** 시도 (기본값)
  - (b) **Simple Icons 기반 키워드 검색** — 기본 아이콘 검색 소스
    - 공식 라이브러리: [Simple Icons](https://simpleicons.org/) (~3,000+ 브랜드 아이콘, SVG, CC0 라이선스)
    - 배포 방식: npm 패키지 `simple-icons`를 앱에 **번들**하여 **오프라인·사내망에서도 동작** (외부 CDN 의존 없음)
    - 검색 매칭: **영문 키워드**로 `title` / `slug` / `aliases` 메타데이터에 대한 부분 일치 (예: "github", "slack"). 한글 입력은 지원하지 않음.
    - 렌더링: SVG를 원본 그대로 표시하며, **브랜드 컬러(hex)** 또는 테마에 맞춘 단색(monochrome) 오버라이드를 선택 가능
    - 사내 전용 시스템 등 Simple Icons에 등재되지 않은 경우는 (a) favicon 또는 (c) 업로드로 폴백
    - (옵션) CDN 실시간 조회(`https://cdn.simpleicons.org/{slug}`)는 **기본 OFF**, 네트워크 정책이 허용할 때 설정으로만 활성화
  - (c) 사용자가 로컬 이미지 파일(.png/.svg/.ico) 업로드
- 아이콘은 로컬에 캐싱되어 오프라인에서도 표시된다.
- Simple Icons 라이브러리는 업데이트 주기가 잦으므로, 앱 버전 릴리스 시 최신 버전으로 **정기 갱신**한다(기대 주기: 분기 1회).

#### FR-3. 편집 모드
- 사용자는 토글 버튼(또는 단축키)으로 편집 모드를 켜고 끈다.
- 편집 모드에서만 다음 조작이 가능해야 한다:
  - **추가**: 섹션 / 카테고리 / 아이템 생성
  - **수정**: 레이블, 대상(URL/파일 경로), 아이콘, 색상/배지 등
  - **삭제**: 확인 다이얼로그 경유
  - **이동**: 드래그 앤 드롭으로 순서 변경 및 계층 간 이동
    - 아이템 ↔ 아이템 (카테고리 내/간)
    - 카테고리 ↔ 카테고리 (섹션 내/간)
    - 섹션 ↔ 섹션 (페이지 내 순서)
- 실행 모드에서는 클릭 시 즉시 실행만 수행하며 편집 UI는 노출되지 않는다.
- **저장 모델 — 확정: 자동 저장 + Undo 스택**
  - 모든 변경은 즉시 디스크에 영속화한다 (NFR 6.2의 원자적 쓰기와 결합, N세대 백업은 후속 과제).
  - 세션 내 Undo/Redo는 Ctrl+Z / Ctrl+Y로 제공하며, 드래그 이동·추가·삭제·수정을 단일 단위로 취급한다.
  - **"데일리 스냅샷"은 앱 구동 시 `config.json` 로드 시점**에 오늘 날짜 파일이 없으면 1회 저장된다(경로: `%APPDATA%\oh-my-workbench\snapshots\YYYY-MM-DD.json`). 최대 7일 보존 후 프루닝. *(PRD 초안은 "편집 모드 진입 시점"이었으나, 구현 단계에서 트리거 지점을 단순화하여 config_store에서 처리)*
  - 구현: Zustand 5 + `zundo` 미들웨어(`temporal`) 사용.

### 5.3 아이템 상세 스펙

**Web URL Item**
- 필수: `label`, `url`
- 선택: `icon`, `browser`(default/edge/chrome/firefox), `browserProfile`, `incognito`(bool), `tags`

**Local App Item**
- 필수: `label`, `target` (파일 경로 또는 AppUserModelID)
- 선택: `icon`, `arguments`, `workingDirectory`, `runAs`(normal/admin — admin은 UAC 경유), `tags`

## 6. 비기능 요구사항 (NFR)

### 6.1 성능
- **앱 콜드 스타트**: 2초 이내 (아이템 300개 기준)
- **아이템 클릭 → 실행 요청**: 200ms 이내
- **편집 중 드래그 프리뷰**: 60fps 유지

### 6.2 신뢰성 / 데이터 무결성
- 저장 중 강제 종료에도 파일 손상이 없도록 **원자적 쓰기**(temp + rename) 적용 — `atomicwrites` crate 사용 (구현 완료)
- 앱 구동 시 config 로드 시점에 **데일리 스냅샷 1회** (FR-6.1 참조, 최대 7일 보존)
- 설정 파일 손상에 대비한 **N세대 자동 백업**은 v1.0에서 **미구현** — 현재는 원자적 쓰기 + 데일리 스냅샷 2중 안전망으로 운용하며, 정식 N세대 백업 회전(`config.json.bak.1 ~ .5`)은 M3 범위로 이월
- **로깅 정책 — 로컬 로그, 외부 송출 없음**
  - 경로: `%APPDATA%\oh-my-workbench\logs\app.log` (v1.0은 `log` crate 기반 초기화만 구현, 본격 로테이션·redact는 M3 범위)
  - 기본 레벨 `info`, `OMWB_LOG=debug` 환경 변수로 `debug` 활성 (예정)
  - **Redact 필수 항목**: URL의 query string, 프로세스 실행 인자 전체 (토큰·비밀번호 포함 가능성) — 구현 시 필수
  - 외부 텔레메트리/크래시 리포팅은 v1.0에서 **도입하지 않음**

### 6.3 보안
- 외부 입력(아이콘 URL, 검색 결과)에 대한 안전한 렌더링 (스크립트 실행 불가, 로컬로 다운로드 후 사용)
- 실행 대상 경로는 사용자가 직접 지정한 것만 허용 — 임의의 네트워크 페이로드로부터 실행 파일이 자동 등록되는 흐름은 금지
- 로그/백업 파일에 **세션 토큰, 자격 증명 평문 저장 금지**
- URL 실행 시 스킴 화이트리스트 (`http`, `https`, `file`, `ms-*` 등) — 이외 스킴은 기본 차단, 사용자 허용 시에만 실행
- **관리자 권한 실행 — 확정: 아이템별 옵션, 기본 OFF**
  - Local App Item의 `runAs` 필드를 `"normal"`(기본) / `"admin"` 중 선택. `"admin"`은 `ShellExecuteW` verb `"runas"`로 실행되어 매 호출 시 Windows UAC 프롬프트를 표시.
  - 설정 UI에 고지 문구 필수: **"이 아이템은 관리자 권한으로 실행됩니다. 실행 시 UAC 프롬프트가 표시됩니다."**
  - 모디파이어 키(Shift-클릭 등)를 통한 임시 승격은 제공하지 않음 — 권한 승격은 항상 명시적 설정 기반.

### 6.4 호환성 및 배포
- Windows 10 (21H2 이상) / Windows 11
- 표준 사용자 계정(관리자 권한 미보유)에서도 설치 및 실행 가능할 것 (Per-user 설치 지원)
- HiDPI / 다중 모니터 지원 (스케일링 100% / 125% / 150% / 200%)
- **배포 — 확정**
  - Tauri 번들러의 **NSIS per-user 설치** (`%LOCALAPPDATA%\oh-my-workbench`)
  - **코드 서명 — v1.0은 무서명**. SmartScreen 경고 1회는 "추가 정보 → 실행"으로 우회. **자동 업데이트(`tauri-plugin-updater`)는 서명 확보 전까지 비활성화**하고, 업데이트는 릴리스 노트 + 수동 재설치 방식으로 운용.
  - 배포 범위가 본인 PC 외로 확장될 경우 **Samsung 사내 코드 서명 인증서** 프로세스를 알아봐 적용. 개인 EV/OV 인증서 구매는 채택하지 않음.

### 6.5 접근성
- 키보드만으로 모든 주요 동작 수행 가능 (Tab 이동, Enter 실행, 화살표 이동, F2 이름 편집 등)
- 포커스 링 및 고대비 모드 준수

### 6.6 국제화
- v1.0: 한국어 우선. 문자열은 리소스 파일로 분리하여 이후 영어 추가 가능하도록 구조화.

## 7. 데이터 모델 (초안)

설정은 단일 JSON 파일로 저장한다. 저장 경로: `%APPDATA%\oh-my-workbench\config.json`.

- `pages[]`는 배열 구조를 유지하지만 **v1.0 UI는 첫 번째 페이지만 렌더링**한다(다중 페이지 UI는 v1.x 확장 지점).
- Local App Item의 `runAs`는 `"normal"`(기본) / `"admin"` 두 값만 허용.

```jsonc
{
  "$schema": "1",
  "settings": {
    "theme": "system",
    "defaultBrowser": "system",
    "globalHotkey": "Ctrl+Alt+Space",
    "startWithWindows": false
  },
  "pages": [
    {
      "id": "page-1",
      "name": "Main",
      "sections": [
        {
          "id": "sec-work",
          "name": "업무 시스템",
          "categories": [
            {
              "id": "cat-portal",
              "name": "사내 포털",
              "items": [
                {
                  "id": "item-xxx",
                  "type": "url",
                  "label": "HR Portal",
                  "url": "https://hr.example.com",
                  "icon": { "kind": "cached", "path": "icons/hr.png" },
                  "browser": "default"
                },
                {
                  "id": "item-yyy",
                  "type": "app",
                  "label": "Visual Studio Code",
                  "target": "C:\\Users\\me\\AppData\\Local\\Programs\\Microsoft VS Code\\Code.exe",
                  "arguments": "",
                  "icon": { "kind": "extracted" }
                }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

- ID는 불변 UUID. 이름 변경이 잦아도 드래그/이동 추적에 문제가 없도록 분리.
- `$schema` 필드로 버전 관리 → 향후 마이그레이션 지원.

## 8. UI / UX 요구사항

### 8.1 레이아웃
- 상단: 제목, 검색 바, 편집 모드 토글, 설정 버튼
- 본문: 섹션들이 세로로 쌓이고, 각 섹션 안에 카테고리 카드가 grid로 배치됨
- 카테고리 카드 안: 아이템 타일이 grid로 배치됨 (타일 크기 2~3단계 조절 가능)

### 8.2 상호작용
- **기본(실행 모드)**: 아이템 클릭 1회 = 실행. 우클릭 = 컨텍스트 메뉴(편집/삭제/복제/속성)
- **편집 모드**: 모든 요소에 드래그 핸들 / 인라인 편집 UI 노출. Esc로 편집 모드 종료.
- **드래그 앤 드롭**:
  - 드롭 타깃 시각화(가이드 라인/하이라이트)
  - 잘못된 드롭(예: 아이템을 섹션 헤더 위로) 시 시각적 거부 피드백

### 8.3 빈 상태 / 온보딩
- 최초 실행 시 예시 섹션/카테고리/아이템(가이드 3~5개)으로 시작하여 구조를 학습할 수 있게 함
- **북마크 가져오기 — 확정: HTML export 파싱 방식, v1.1 범위**
  - 사용자가 Edge/Chrome에서 "즐겨찾기 HTML 내보내기"로 생성한 Netscape 포맷 파일을 선택하여 가져오기
  - 브라우저 프로필 디렉터리(`%LOCALAPPDATA%\...\User Data\...\Bookmarks`)의 **직접 읽기는 채택하지 않음** — 사내 DLP/EDR 오탐 회피 및 최소 권한 원칙
  - 가져온 폴더 트리는 섹션/카테고리로 매핑하되, 기존 구성과의 병합 전략은 설계 단계에서 상세화

## 9. 기술 스택 (확정, 2026-04-21)

### 9.1 결정
**Tauri 2.x** — Rust 백엔드 + Web 프론트엔드.

### 9.2 선정 근거
- **경량성**: 설치 파일 3~10MB, RAM 30~80MB, 콜드 스타트 <1s — 본 프로젝트가 지향하는 "런처" 특성에 부합하며 NFR 성능 목표에 여유를 둠
- **네이티브 연동**: Rust `windows` crate를 통해 Win32 / WinRT 전체 API 접근 가능. 아이콘 추출·UWP(AppUserModelID)·UAC·레지스트리 조회 등 WPF 수준의 통합 가능
- **UI 반복 속도**: React + Tailwind + dnd-kit 등 웹 스택의 생산성 활용 — 편집 모드·드래그 앤 드롭·테마 구현 유리
- **런타임 의존성 최소화**: Windows 10 21H2+ / 11에 기본 포함된 WebView2 사용 (Chromium 번들 불필요)
- **보안**: 기본 CSP·capability 기반 권한 모델이 엄격 — 런처처럼 실행 권한을 트리거하는 앱의 공격 표면 축소에 유리

### 9.3 확정 구성

| 영역 | 선택 | 비고 |
|---|---|---|
| 앱 셸 | Tauri 2.x | — |
| 프론트엔드 빌드 | Vite + TypeScript | — |
| UI 프레임워크 | React | — |
| 스타일 | Tailwind CSS | 다크/라이트 테마 구현 용이 |
| 상태 관리 | Zustand | 경량, 러닝 커브 낮음 |
| 드래그 앤 드롭 | dnd-kit | 접근성·중첩 컨테이너 지원 |
| 아이콘 라이브러리 | `simple-icons` npm 패키지 (번들) | FR-2 참조 |
| 백엔드 언어 | Rust (stable) | — |
| Windows API | `windows` crate | Win32 / WinRT |
| 직렬화 | `serde` / `serde_json` | — |
| 원자적 쓰기 | `atomicwrites` crate | NFR 6.2 대응 |
| 비동기 런타임 | `tokio` (기본 feature만) | — |
| Tauri 플러그인 | `global-shortcut`, `updater`, `fs`, `dialog`, `shell` | FR-11, 자동 업데이트, 파일 I/O |

### 9.4 모듈 경계 (상위 수준)
- **프론트엔드(renderer)**: UI, 상태, 편집 로직, Simple Icons 검색/렌더링, 설정 스키마 validation
- **백엔드(Rust core)**: 파일 I/O(원자적 쓰기), 아이콘 추출(`.exe`/`.lnk`/UWP), 프로세스 실행(인자·workdir·UAC), 브라우저 프로파일 탐색, 전역 단축키, 트레이
- **경계**: Tauri `#[command]` — 입력은 JSON 직렬화 값, 반환도 동일. 렌더러는 임의의 파일/프로세스에 직접 접근하지 않음 (FR-NFR 6.3 보안 요건 충족).

# Claude Code로 구현한 업무 자원 통합 런처 'oh-my-workbench' 개발 사례

---

## 1. 목적 / 배경

### 문제 상황

일상 업무에서 사용하는 자원이 여러 경로에 분산되어 있어 접근 비용이 누적되고 있었다.

- 사내 웹 시스템, SaaS, 외부 서비스 URL → Edge/Chrome 북마크 폴더
- 자주 쓰는 로컬 프로그램 → 작업 표시줄 고정 아이콘, 바탕화면 바로가기
- 카테고리 분류 → 브라우저 북마크 폴더와 탐색기 폴더에 이중 관리

"이 도구 어디 있더라?"를 탐색하는 시간이 매 업무마다 발생하고, PC를 교체하거나 브라우저를 바꿀 때마다 동일한 구성을 처음부터 다시 작업해야 했다. 또한 웹 URL과 로컬 앱이 서로 다른 공간에 존재하여, 하나의 화면에서 업무 흐름 전체를 조망할 수 없었다.

### 해결 목표

모든 업무 자원(웹 URL + 로컬 프로그램)을 카테고리별로 **한 페이지에서 시각적으로 파악**하고, **단 한 번의 클릭으로 실행**할 수 있는 개인용 Windows 런처 애플리케이션을 개발하는 것을 목표로 삼았다. 추가로 JSON 파일 기반의 설정 내보내기/가져오기를 통해 PC 이전 시에도 동일한 환경을 즉시 복원할 수 있도록 하였다.

## 2. GenAI 활용 내용 및 수행 내용

### 활용 도구

삼성 Gen AI 서비스를 통해 제공된 **Claude Code(CLI)** 를 주 개발 도구로 사용하였다. Claude Code는 Anthropic의 Claude 모델을 기반으로 한 터미널 통합 개발 보조 도구로, 코드 작성·수정·검토를 대화형으로 수행할 수 있다.

### 수행 과정

**1단계 — 기획 및 설계 문서 작성**

Claude Code와의 대화를 통해 PRD(제품 요구사항 명세서), ARCHITECTURE(기술 아키텍처 설계서), UX(화면/인터랙션 정의서) 세 가지 문서를 순차적으로 작성하였다. 기획 의도와 제약 조건을 자연어로 설명하면 Claude Code가 구조화된 마크다운 문서로 정리하고, 빠진 결정 사항(기술 스택 선택, 저장 전략, 보안 정책 등)을 질의하여 함께 확정하는 방식으로 진행하였다.

확정된 주요 기술 스택:
- **앱 프레임워크**: Tauri 2.x (Rust 백엔드 + React/TypeScript 프론트엔드)
- **UI**: React 19, Tailwind CSS 4, dnd-kit (드래그 앤 드롭)
- **상태 관리**: Zustand 5 + zundo (Undo/Redo 지원)
- **Windows 연동**: 직접 FFI (ShellExecuteW, SHGetFileInfoW) via shell32.dll, gdi32.dll
- **아이콘**: Simple Icons CDN (웹 URL), Windows 아이콘 추출 (로컬 앱)

**2단계 — 기능별 수직 슬라이스 구현**

설계 문서를 기반으로 기능 단위(FR)별로 구현을 진행하였다. 각 기능마다 Rust 백엔드 command → 프론트엔드 IPC 래퍼 → UI 컴포넌트 순서로 end-to-end 구현 후 동작을 직접 확인하며 다음 기능으로 넘어가는 방식을 택했다.

구현된 주요 기능:

| 기능 | 내용 |
|---|---|
| 아이템 렌더링 (FR-1) | 섹션 → 카테고리 → 아이템 계층 구조, 타일 그리드 레이아웃 |
| 아이콘 처리 (FR-2) | Simple Icons CDN 연동, Windows exe 아이콘 추출 (SHGetFileInfoW → GDI → PNG → base64) |
| 편집 모드 (FR-3) | 섹션/카테고리/아이템 추가·수정·삭제, dnd-kit 드래그 앤 드롭, Undo/Redo |
| 전역 검색 (FR-4) | Ctrl+K 단축키, 레이블·URL·경로 인덱싱, 섹션/카테고리 breadcrumb 표시 |
| 내보내기/가져오기 (FR-6) | 편집 모드 전용 버튼, 원자적 파일 쓰기, 내보내기 완료 토스트 알림 |
| 다크/라이트/시스템 테마 (FR-7) | `<html class="dark">` 토글, Tailwind CSS 4 `@variant dark` |
| 전역 단축키 (FR-11) | tauri-plugin-global-shortcut, 창 show/hide 토글, 설정 UI에서 단축키 변경 |
| 데일리 스냅샷 | 설정 파일 로드 시 `snapshots/YYYY-MM-DD.json` 자동 저장, 7일 보존 |
| 아이템별 아이콘 스타일 | URL 아이템별 auto/light/dark 옵션 (CSS 필터 기반) |

**3단계 — 대화형 검증 및 반복 수정**

각 기능 구현 후 사용자가 직접 실행하여 확인하고, 발견된 문제나 추가 요구사항을 자연어로 전달하면 Claude Code가 관련 파일을 읽고 수정하는 방식으로 반복하였다. 예를 들어 "내보내기 후 사용자에게 알림이 없다", "다크모드에서 아이콘이 보이지 않는다", "편집 모드에서만 내보내기 버튼이 활성화되어야 한다" 등의 피드백을 그대로 입력하면 즉시 코드 수정이 이루어졌다.

Rust 컴파일 에러와 dead-code 워닝도 터미널 출력을 그대로 붙여넣으면 원인 분석 후 수정 방안을 제시하고 파일을 직접 편집하는 방식으로 해결하였다.

## 3. 시사점 / Lessons Learned / 향후 계획

### 얻은 인사이트

**자연어 기획이 실제 코드로 직결되는 경험**이 가장 큰 인사이트였다. 기존에는 아이디어를 기획 문서로 정리하고, 그것을 다시 기술 명세로 변환하고, 개발자가 구현하는 3단계를 거쳤다. Claude Code를 통해 기획 의도를 바로 설계 문서로, 설계 문서를 바로 코드로 이어가는 흐름이 가능함을 확인했다. 비개발 직군도 아이디어 수준에서 실제 동작하는 프로토타입까지 도달하는 시간이 획기적으로 단축될 수 있다.

**Rust + Windows Win32 FFI**처럼 학습 비용이 높은 영역에서도 Claude Code가 올바른 unsafe 코드와 Windows API 사용 패턴을 제시하여 직접 구현을 주도할 수 있었다. 단, 생성된 코드의 보안 및 안전성은 개발자가 최종 검토해야 함을 확인하였다.

### 아쉬운 점 및 한계

- **컨텍스트 한계**: 장기 개발 세션에서 이전 대화 내용이 점진적으로 압축되어, 초반에 결정한 설계 세부 사항이 후반 구현에 반영되지 않는 경우가 발생하였다. CLAUDE.md나 아키텍처 문서를 통해 핵심 결정을 지속적으로 참조할 수 있도록 하는 것이 중요하다.
- **UI 동작 검증의 한계**: Claude Code는 코드를 작성하지만 렌더링된 화면을 직접 볼 수 없다. 타입 체크·빌드 성공이 곧 기능 정상 동작을 의미하지 않으므로, 각 기능 구현 후 반드시 직접 실행 확인이 필요하다.
- **미구현 기능**: PRD에 정의된 기능 중 실행 이력 기반 정렬(FR-5), 기본 브라우저 지정(FR-8), 시작 시 자동 실행(FR-12), 복수 페이지/탭(FR-10), HTML 북마크 가져오기(v1.1) 등은 아직 미구현 상태다.

### 향후 계획

- **업무 현장 실적용**: 실제 사내 시스템 URL과 자주 사용하는 프로그램을 등록하여 일상 업무에 투입하고, 불편한 점을 수집하여 v1.1 우선순위에 반영한다.
- **HTML 북마크 가져오기 (v1.1)**: 기존에 브라우저에 정리해 둔 북마크를 한 번에 마이그레이션할 수 있는 기능을 추가하여 초기 설정 비용을 낮춘다.
- **팀 단위 확장 가능성**: 현재는 개인용이지만, 팀 공통 자원(팀 위키, 모니터링 대시보드, CI/CD 도구 등)을 공유 config.json으로 배포하는 팀 런처로의 확장을 검토할 수 있다.
- **Claude Code 활용 범위 확장**: 이번 경험을 바탕으로 업무 자동화 스크립트, 사내 데이터 분석 도구 등 다른 생산성 도구 개발에도 Claude Code를 적극 활용할 계획이다.
